curl -sS https://api.yanylevin.com/health

Example shape:

  "service": "yanylevin-server",
  "authConfigured": true,
  "jwt": { "issuer": "yanylevin-next", "audience": "yanylevin-mac-api" },

If lmStudio.modelLoaded is false, chat will 502 until LM Studio is started on the Mac.

POST /api/chat — requires Bearer JWT

POST https://api.yanylevin.com/api/chat
Authorization: Bearer <jwt>
Content-Type: application/json

{
  "messages": [
    { "role": "system", "content": "…" },
    { "role": "user", "content": "Who is Yan?" }
  ],
  "temperature": 0.4,
  "maxTokens": 2048
}
FieldRules

temperatureOptional number (default 0.4)
maxTokensOptional number (default 2048)

Auth gap (frontend must solve this)

/api/chat is protected with HS256 JWT:

  • Issuer: yanylevin-next
  • Audience: yanylevin-mac-api
  • Secret: AUTH_SECRET in server/.env on the Mac
  • Required claim: email (or sub containing @)

Pattern from NoteLMs / SocketHR: browser calls a same-origin Vercel route (e.g. GET /api/mac-token) which mints the JWT with the shared secret, then the browser calls api.yanylevin.com with Authorization: Bearer ….

Today the site is plain static HTML on Vercel — there is no Next.js / Auth.js / mac-token route yet. You cannot put AUTH_SECRET in client JS.

Recommended auth options (pick one; do not invent a fourth unless necessary)

  1. A — Minimal public FAQ bot (likely best for a personal site)
    Add a tiny Vercel serverless route (can keep static HTML elsewhere, or migrate just api/): rate-limited /api/mac-token that mints a short-lived JWT for a fixed visitor identity (e.g. visitor@yanylevin.com) using AUTH_SECRET / NEXTAUTH_SECRET set in Vercel env to the exact same string as Mac server/.env. No Google login required. Add abuse controls (IP rate limit, short TTL ~5–10 min).
  2. B — NoteLMs clone (Google sign-in)
    Migrate site (or a subdirectory app) to Next.js + Auth.js Google OAuth. Copy NoteLMs web/src/pages/api/mac-token.ts + mintMacToken pattern; change issuer/audience to yanylevin-next / yanylevin-mac-api. Heavier than needed unless you want signed-in-only chat.
  3. C — Same-origin chat proxy
    Browser posts to https://yanylevin.com/api/chat (Vercel function); that function holds the secret, mints/uses JWT (or calls Mac with a server-side token), and proxies to api.yanylevin.com. Browser never sees the Mac JWT. Slightly more latency; good if you want to hide Mac auth details.

Mac secret lives in /Users/yanlevin/github/yanylevin/server/.env (gitignored). When you add Vercel env, set AUTH_SECRET or NEXTAUTH_SECRET to match it, then redeploy. Never commit the secret.

Knowledge base for the bot

Canonical facts file: data/yan.md. It already instructs the model: prefer facts from the file; if unknown, say so.

Frontend responsibility: on each chat (or once per session), send a system message that includes (or summarizes) data/yan.md, plus tone/safety instructions. The Mac API does not auto-load yan.md today.

Options: (1) fetch /data/yan.md from the static site and stuff into system prompt; (2) bake a trimmed system prompt into the JS bundle; (3) later add a Mac endpoint that injects yan.md server-side (backend change — only if needed).

Suggested frontend implementation steps

  1. Confirm backend: curl -sS https://api.yanylevin.com/health (expect service: "yanylevin-server").
  2. Choose auth option A/B/C above. Implement same-origin token or proxy first — chat UI is useless without it.
  3. Add a small runtime config so production uses https://api.yanylevin.com and local preview can override (NoteLMs uses public/runtime-config.json + optional env). Example: {"apiBase":"https://api.yanylevin.com"}.
  4. Build chat UI on the existing site (panel / route / section — keep brand first). Client helper roughly: 
    const tokenRes = await fetch("/api/mac-token", { credentials: "same-origin" });
const { token } = await tokenRes.json();
const res = await fetch(`${apiBase}/api/chat`, {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    Authorization: `Bearer ${token}`,
  },
  body: JSON.stringify({ messages }),
});
  5. Prefill messages[0] as system with yan.md + “answer about Yan; if unknown, say so”. Keep conversation history in memory; send last N turns (API already caps at 40).
  6. Handle offline Mac / unloaded model: if health shows modelLoaded: false or chat returns 502, show a calm offline message (do not spin forever).
  7. Push to main. Verify on production origin (CORS + real tunnel), not only localhost.

UX / design constraints

  • Site already has a strong personal brand (Bricolage Grotesque + Karla, hero portrait). Match it — do not drop in a generic purple AI chat widget.
  • Prefer one clear chat entry (nav item, floating control, or section) — not a dashboard of AI chrome.
  • Follow repo frontend-design rules: avoid Inter/Roboto defaults, avoid purple-glow AI clichés, avoid packing the hero with chat UI.
  • Mobile: chat must work on small screens; site already has responsive layout.
  • Accessibility: focus trap in modal if used; Esc to close; labels on inputs.
