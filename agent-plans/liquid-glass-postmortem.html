1. Original goal

  • Implement a material identical to Apple Liquid Glass (WWDC 2025 / iOS 26 language).
  • Research GitHub / web demos that already solved this.
  • Add two fixed floating elements that stay on screen while scrolling and sit above page content:
    • Bottom left: Apple squircle
    • Bottom right: circle
  • Push to main.

The user later reported no visible warping of content behind the orbs, and then that the implementation “looks nothing like liquid glass.” That feedback is correct relative to Apple’s product UI.

2. What shipped (current state)

Two fixed corner orbs with layered CSS (tint / inset shine / specular gradient) plus a JS module that tries to build physics-ish SVG displacement maps and apply them via backdrop-filter: url(#…) on Chromium, with a turbulence filter: url(#…) fallback elsewhere.

PieceLocationRole
Markup index.html (near end of body) Two .liquid-glass-orb nodes + empty SVG #liquid-glass-defs
Engine liquid-glass.js IOR bevel profile, canvas displacement/specular maps, filter injection, browser branching
Styles styles.css (bottom “Apple Liquid Glass floating orbs” section) Fixed positioning, shape radii, frost layers, float animation
What users actually see: soft translucent circles with white edge highlights — classic glassmorphism. Little or no convincing lens refraction, no Apple-like “liquid” motion, no material response that reads as iOS 26 chrome. On Safari, true SVG backdrop displacement does not run at all.

4. Method the prior agent attempted

Intended pipeline (Chromium)

  1. Compute a 1D refraction profile from glass thickness, bezel width, IOR, and a surface height function (convex_circle / convex_squircle).
  2. Rasterize a displacement map (R=X shift, G=Y shift) and a specular map on canvas.
  3. Inject an SVG <filter> with feGaussianBlur → feImage (maps) → feDisplacementMap → specular composites.
  4. Apply backdrop-filter: blur(…) url(#filterId) … on .lg-effect.

Fallback pipeline (Safari / Firefox)

  1. Plain backdrop-filter: blur + saturate + brightness.
  2. Additionally filter: url(#lg-turbulence-distort) to warp the frosted bitmap (not true page refraction).

Shapes

  • Circle: border-radius: 50%
  • Squircle: border-radius: 28% + corner-shape: squircle (where supported), plus a superellipse (n≈5) attempt in the displacement generator

5. Mistakes (avoid these)

Mistake A — Declared victory without visual proof

The agent spent time on research and plumbing, then pushed to main without a hard visual comparison against Apple UI or even against archisvaze/liquid-glass / Lucas demos side-by-side. When the user said it looked wrong, that should have been treated as a hard fail, not a messaging problem.

Rule: If you cannot open Chrome, overlay the orb on busy content, and clearly see lens-like edge refraction matching a reference demo, you are not done.

Mistake B — Backdrop roots killed refraction

First CSS pass put isolation: isolate (and later transform animation / will-change: transform) on the orb wrapper. That creates a backdrop root: the effect samples empty pixels inside the orb instead of the page. Result: tinted frost, zero meaningful warp. Partially fixed later by removing isolation and switching float animation to margin-bottom, but this cost a full user complaint cycle. 

/* BAD — do not put these on an ancestor of the backdrop-filter layer */
isolation: isolate;
transform: …;          /* including animated transforms */
will-change: transform;
opacity: < 1 on wrapper;
filter: … on wrapper;

Mistake C — Treated “CSSOM accepts url(#test)” as “Safari works”

Safari can accept backdrop-filter: url(#…) in the style object and still not render SVG reference filters for backdrop. UA sniffing was added late. Assume Chromium-only for true refraction unless you verify otherwise on a real Safari build.

Mistake D — Fragile SVG filter injection

First version used insertAdjacentHTML into SVG <defs>, which can create HTML-namespaced nodes that fail as filters. Later switched to parsing via an SVG holder — better, but still never validated that the filter actually displaced pixels in the live page.

Mistake E — Wrong visual recipe even when refraction “works”

Apple Liquid Glass is not only edge displacement. The prior agent under-delivered on:

  • Convincing lens thickness / rim light (product-quality specular, not a soft CSS radial)
  • Material that feels continuous with OS chrome (controls, sheets, buttons) rather than decorative blobs
  • Motion: subtle liquid response, not a bobbing margin float
  • Contrast against this site’s calm gradient background — refraction needs busy/high-frequency content behind the glass to read; corner placement over empty footer-ish areas hides the effect
  • Correct blur budget: Apple glass is often sharper with localized bend; dumping heavy blur washes the lens look into frost

Mistake F — Over-adapted physics code, under-matched reference demos

Copying archis math without also copying their full visual stack (shadows, tint opacity, specular opacity, scale, background choice, element size, and live tuning UI) produced a pale imitation. The reference demos look good partly because of scene (rich wallpaper) and carefully tuned defaults — not the formula alone.

Mistake G — Shipped decorative orbs instead of a reusable material

The ask was a material identical to Apple’s. The delivery was two fixed ornaments. Even if the filter were perfect, there is no shared primitive usable for nav/buttons/cards, and no proof the material holds up at different sizes.

Mistake H — Process / tooling failures

  • Got stuck attempting screenshots / browser tooling instead of finishing validation another way (user had to interrupt).
  • Pushed incomplete visuals to main early, then patched reactively.
  • Did not leave an agent plan document until the user asked for this one.

7. Browser reality check

  • Chromium Only reliable target today for backdrop-filter: url(#svgFilter) with feDisplacementMap / feImage.
  • Safari SVG reference filters on backdrop are dropped (as of this writing). Frost + optional filter:url on the layer only.
  • Firefox Same class of limitation for rich SVG backdrop graphs.

If the user is on Safari (likely, given Apple material ask), they will never see the Chromium refraction path. Either ship a WebGL/shader path that works cross-browser (archis has webgl.html), or set expectations explicitly — but still make Chromium look actually like the reference.

9. Recommended playbook for the next agent

  1. Pick a known-good reference and match it first offline. Start from archisvaze/liquid-glass (SVG and consider WebGL) or Lucas’s CodeSandbox. Reproduce one circle and one squircle on a busy background page until they are indistinguishable from the demo at default “Apple-like” settings.
  2. Validate in Chrome with busy content under the glass. Put the orb over the portrait and over text. You must see rim bend. If not, stop integrating into the marketing site.
  3. Keep the backdrop-filter element free of backdrop roots. No isolation, opacity, filter, or transform on ancestors of the sampling layer. Prefer animating position properties that do not create containing/backdrop roots, or animate a sibling chrome layer.
  4. Prefer WebGL if “identical to Apple” is non-negotiable cross-browser. SVG backdrop displacement will never satisfy Safari users. Archis’s WebGL path exists for this reason.
  5. Tune like a product material, not a math toy. Match: outer shadow, inner light wrap, tint (~few % white), bezel width as a fraction of radius, low blur, high localized displacement at rim, specular map opacity. Steal numbers from the reference demo’s defaults before inventing your own.
  6. Scene matters. This site’s soft mint/burgundy wash is a bad showcase. Temporarily add a high-frequency test backdrop (or force orbs over the portrait) while developing; then restyle for production once the lens is undeniable.
  7. Only then wire into index.html. Fixed bottom-left squircle / bottom-right circle, position: fixed, high z-index, pointer-events as product requires.
  8. Hard-refresh / cache-bust JS when iterating (liquid-glass.js?v=…) so you are not debugging a stale file.

10. Acceptance criteria (pass/fail)

Do not merge/push as done unless all of these pass:

  • Pass Side-by-side with archis/Lucas (or Apple OS screenshot): rim refraction quality is in the same league in Chrome.
  • Pass Over the site portrait, text letters visibly bend near the glass edge while scrolling.
  • Pass Squircle reads as Apple continuous-corner / squircle, not a rounded square.
  • Pass Orbs remain fixed bottom-left / bottom-right above content while scrolling.
  • Pass Safari has an intentional fallback that still looks premium (or WebGL works there) — not a milky blank disk.
  • Fail “It’s blurry and shiny” without lens distortion.
  • Fail Effect only visible on a wallpaper demo page, disappears on this site.

11. Explicit do-not list

  • Do not call blur + saturate + inset highlight “Liquid Glass.”
  • Do not put isolation: isolate on the glass wrapper.
  • Do not animate transform on a backdrop-filter ancestor.
  • Do not trust Safari CSSOM acceptance of url(#filter).
  • Do not push to main until Chrome shows obvious refraction.
  • Do not spend the session stuck on screenshot infrastructure; use a real browser check or a minimal headless dump only if it unblocks comparison quickly.
  • Do not treat /2/index.html as an Apple-identical reference.
  • Do not increase tint/opacity to “make it more visible” — that makes it less like Apple glass.
